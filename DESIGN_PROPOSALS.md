# 超级技能栏框架设计方案

## 需求总结
1. **完整复刻SkillWnd**：不是简单按钮+面板，而是完整的技能窗口系统
2. **吸附在SkillWnd左侧**：类似宏编辑器吸附在右侧的方式
3. **同层级显示**：与SkillWnd同层，不是最上层
4. **完整功能**：技能图标、拖拽、点击使用、页签切换等
5. **独立管理**：可自由添加/修改技能，不依赖游戏原生技能数据

---

## 方案1：完全复刻SkillWndEx类（最接近原生）

### 架构
```
SuperSkillWnd (0xAE8 bytes)
  ├─ 继承CWnd基类布局 (0~131)
  ├─ 技能列表数组 (this+697)
  ├─ ListView控件 (this+699)
  ├─ Tab页签控件 (this+701)
  ├─ 滚动条控件 (this+717)
  └─ 自定义技能数据结构
```

### 优点
- 与游戏原生结构100%兼容
- 可直接复用游戏的ListView/Tab/ScrollBar控件
- 消息路由、HitTest、Draw完全一致
- 最稳定，最不容易出bug

### 缺点
- 需要完整实现2792字节的对象布局
- 需要Hook游戏的控件创建函数
- 复杂度最高

### 关键技术点
1. 复制SkillWndEx的vtable (off_E66F50)
2. 调用游戏的ListView创建函数 (sub_9E17D0)
3. 实现技能数据结构 (替代this+697)
4. 父子关系：设置parent为SkillWnd，相对坐标(-174, 0)

---

## 方案2：轻量级CWnd + 游戏控件（推荐）

### 架构
```
SuperSkillWnd (简化CWnd)
  ├─ CWnd基类 (132 bytes)
  ├─ 借用SkillWnd的ListView控件
  ├─ 自定义Draw函数
  └─ 自定义技能数据管理
```

### 优点
- 结构简单，只需132字节基类
- 借用游戏控件，减少重复实现
- 可灵活定制Draw逻辑
- 开发速度快

### 缺点
- 需要手动管理控件生命周期
- 可能需要Hook控件的parent检查

### 关键技术点
1. 创建基础CWnd (sub_B9BF60 + sub_A996B0 + sub_A99330)
2. 设置parent为SkillWnd (this+parent_offset)
3. 借用SkillWnd的ListView/ScrollBar
4. 自定义Draw函数渲染技能图标

---

## 方案3：纯D3D9渲染 + 自定义HitTest（最灵活）

### 架构
```
SuperSkillWnd (最小CWnd)
  ├─ CWnd基类 (仅用于占位和消息路由)
  ├─ 纯D3D9纹理渲染
  ├─ 自定义HitTest逻辑
  └─ 自定义技能数据 + 拖拽系统
```

### 优点
- 完全自主控制渲染
- 不依赖游戏控件
- 可实现任意UI效果
- 最灵活

### 缺点
- 需要自己实现所有交互逻辑
- 拖拽、滚动、页签都要自己写
- 与游戏风格可能不一致

### 关键技术点
1. 最小CWnd (只用于占位)
2. Present中D3D9渲染所有内容
3. WndProc中自定义HitTest
4. 实现拖拽系统 (WM_LBUTTONDOWN/MOVE/UP)

---

## 方案4：子窗口数组（模块化）

### 架构
```
SuperSkillWnd (容器CWnd)
  ├─ 背景CWnd (174x299)
  ├─ Tab按钮CWnd数组 (每个页签一个)
  ├─ 技能图标CWnd数组 (每个技能一个)
  └─ 滚动条CWnd
```

### 优点
- 模块化，每个组件独立
- 易于维护和扩展
- 符合游戏的子控件系统

### 缺点
- 需要创建大量CWnd对象
- 内存占用较大
- 管理复杂度高

### 关键技术点
1. 创建容器CWnd
2. 为每个组件创建子CWnd
3. 设置子控件链表 (this+28)
4. 实现子控件的Draw/HitTest

---

## 方案5：混合方案（平衡）

### 架构
```
SuperSkillWnd (简化SkillWndEx)
  ├─ CWnd基类 (132 bytes)
  ├─ 扩展字段 (200 bytes)
  │   ├─ 技能数据数组
  │   ├─ 当前页索引
  │   └─ 滚动偏移
  ├─ 游戏原生Tab控件 (借用)
  ├─ 游戏原生ScrollBar (借用)
  └─ 自定义Draw渲染技能图标
```

### 优点
- 结构清晰，大小适中 (332 bytes)
- 复用游戏控件（Tab/ScrollBar）
- 自定义技能渲染
- 开发难度适中

### 缺点
- 需要部分Hook控件创建
- 需要理解游戏控件接口

### 关键技术点
1. 创建332字节对象
2. 借用Tab/ScrollBar控件
3. 自定义Draw函数
4. 实现技能数据管理

## 最终推荐：方案2（轻量级CWnd + 游戏控件）

### 理由
1. **开发效率高**：只需实现核心逻辑，控件复用游戏原生
2. **兼容性好**：使用游戏的控件系统，不容易出问题
3. **灵活性足够**：可以自定义技能数据和渲染
4. **维护简单**：代码量适中，易于调试

### 核心实现步骤
1. 创建基础CWnd (132 bytes)
2. 设置为SkillWnd的子窗口
3. 借用SkillWnd的资源（字体、图标句柄）
4. 自定义Draw函数渲染技能列表
5. 实现WndProc处理点击/拖拽
6. 实现技能数据管理系统

## 关键问题解决方案

### 问题1：按钮无反应
**原因**：消息路由被父窗口拦截
**解决**：
1. 检查z-order (this+32)，确保不被遮挡
2. 注册到可交互链表 (dword_F57420)
3. 实现正确的HitTest (vtable+24)

### 问题2：在最上层显示
**原因**：z-order设置过高
**解决**：
1. 设置z-order = SkillWnd的z-order (相同层级)
2. 使用sub_BA1E80插入到z-order链表
3. 确保parent设置正确

### 问题3：位置微调
**原因**：当前计算有偏差
**解决**：
```cpp
// 当前：swX - 174, swY
// 修正：swX - 174 - 5, swY - 5
int panelX = swX - EXPANDED_W - 5;
int panelY = swY - 5;
```

### 问题4：吸附关系
**原因**：没有设置父子关系
**解决**：
1. 设置parent指针 (this+parent_offset)
2. 使用相对坐标而不是绝对坐标
3. 监听SkillWnd的移动/关闭事件
